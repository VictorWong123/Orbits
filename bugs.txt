SEVERITY: HIGH

[H-1] deleteProfile bypasses authenticated Supabase client (Security)
  File:     backend/actions/profiles.ts — lines 52-64
  Issue:    `deleteProfile` calls `createClient()` directly instead of
            `getAuthenticatedSupabase()`. As a result:
            (a) There is no explicit authentication gate before the delete executes.
            (b) The delete query has no `.eq("user_id", user.id)` filter, unlike
                `deleteFact` and `deleteEvent` which both add this as defense-in-depth.
            RLS is the only protection. If RLS were ever misconfigured this would
            allow unauthenticated deletion of any profile by ID.
  Fix:      Replace `createClient()` with `getAuthenticatedSupabase()` and add
            `.eq("user_id", user.id)` to the delete query. Return "Not authenticated"
            if auth is null, consistent with every other write action in the codebase.
  Example fix:
    const auth = await getAuthenticatedSupabase();
    if (!auth) return "Not authenticated";
    const { supabase, user } = auth;
    const { error } = await supabase.from("profiles").delete()
      .eq("id", profileId).eq("user_id", user.id);

[M-1] migrateLocalData accepts arbitrary user-supplied data without validation (Security)
  File:     backend/actions/migration.ts — lines 25-56
  Issue:    The action accepts a `LocalSnapshot` payload from the browser and inserts
            it into Supabase with no field-level validation (no length caps, no type
            checks on content strings). A crafted request could insert profiles with
            extremely long `full_name` strings, facts with very long `content`, etc.
            Additionally, the `id` fields from local storage are trusted verbatim as
            UUID primary keys — a colliding UUID would cause a silent insert failure.
  Fix:      Add a Zod schema that validates each profile, fact, and event row before
            insertion. At minimum: cap `full_name` ≤ 200 chars, `content` ≤ 5000 chars,
            `title` ≤ 500 chars. Also validate that all `id` fields are valid UUIDs.
  Severity notes: Medium because client data goes server-to-Supabase and RLS still
            enforces user_id ownership. The risk is data quality / DoS, not auth bypass.

[M-2] updateSettings does not validate paletteId against known values (Security)
  File:     backend/actions/settings.ts — lines 41-53
  Issue:    `updateSettings(paletteId)` accepts any arbitrary string and stores it in
            `user_settings.palette_id`. There is no check that `paletteId` is one of
            the six defined palette IDs ("sage", "ocean", etc.). A malicious actor
            could write arbitrary strings to the database.
  Fix:      Add a Zod schema: `z.enum(["sage", "ocean", "lavender", "rose", "amber", "slate"])`.

[M-3] SupabaseDataStore.deleteFact and deleteEvent missing user_id defense-in-depth (Security)
  File:     frontend/lib/store/SupabaseDataStore.ts — lines 162-165, 194-197
  Issue:    Both client-side delete methods filter only by `id`, without also filtering
            by `user_id`. The server-action counterparts (`backend/actions/facts.ts`,
            `backend/actions/events.ts`) both add `.eq("user_id", user.id)` as
            defense-in-depth. The client store does not, creating an inconsistency.
  Fix:      Fetch `userId` via `this.getUserId()` and add `.eq("user_id", userId)` to
            both delete queries (same pattern as `deleteFact` in the server action).

[M-4] importSharedCard does not deduplicate — same card can be imported multiple times
  File:     backend/actions/cards.ts — lines 113-163
  Issue:    A user can call `importSharedCard` for the same `cardId` multiple times,
            creating duplicate profile rows (all with the same `imported_data.card_id`).
            There is no uniqueness check before inserting the new profile.
  Fix:      Before inserting, query for an existing profile where
            `imported_data->>'card_id' = cardId` and `user_id = user.id`. Return an
            informative error like "You've already imported this card." if found.

[M-5] createCard does not validate email format (Security / Data Quality)
  File:     backend/actions/cards.ts — line 15
  Issue:    The `email` field in `CreateCardSchema` is only validated for max length
            (254 chars), not for email format. Any string (e.g. "notanemail") passes
            validation and is stored in the database.
  Fix:      Change the schema to:
              email: z.string().email("Invalid email format").max(254).optional()

[M-6] getFriends and getReminders make N+1 Supabase RPC calls (Performance)
  File:     frontend/lib/store/SupabaseDataStore.ts — lines 253-287, 372-391
  Issue:    `getFriends` fetches all friendship rows, then for each friend fires two
            parallel RPCs (`get_user_email_by_id` + `get_user_profile_by_id`). With N
            friends that's 1 + 2N round-trips. `getReminders` makes 1+N round-trips
            (one per sender email lookup).
            For users with 10 friends: 21 RPC calls on every page load.
  Fix:      Create batched PostgreSQL functions (e.g. `get_friends_with_profiles(user_id)`)
            that return all needed data in a single query. Alternatively, use a Supabase
            `.select()` with embedded joins if the RLS policies allow it.

[M-7] MigrationModal does not handle partial server-side migration failures (Bug)
  File:     frontend/components/ui/MigrationModal.tsx — lines 35-53
            backend/actions/migration.ts — lines 40-53
  Issue:    `migrateLocalData` inserts profiles → facts → events sequentially. If
            profiles succeed but facts fail, partial data is in Supabase. The modal
            surfaces the error string but leaves localStorage intact. On a retry, the
            already-inserted profiles will trigger a "duplicate key" error, making the
            migration permanently broken without manual intervention.
  Fix:      Wrap all three inserts in a single Supabase RPC that executes in a
            PostgreSQL transaction. If any insert fails the whole batch rolls back.
            Alternatively, use `upsert` with `onConflict: "id"` for all three inserts
            so retries are idempotent.

[L-1] orbits_palette cookie is not set with Secure flag (Security / Best Practice)
  File:     frontend/lib/store/LocalDataStore.ts — line 250
            frontend/lib/store/SupabaseDataStore.ts — line 230
  Issue:    Both stores write `document.cookie = "orbits_palette=...;SameSite=Lax"`
            without the `Secure` attribute. The cookie will be sent over HTTP in
            non-HTTPS environments. (Low risk since the value is non-sensitive.)
  Fix:      Append `;Secure` to the cookie string, or better yet centralise cookie
            writing into a shared utility that conditionally adds Secure in production.

[L-2] ConfirmDialog autoFocus is on the destructive button (UX / Safety)
  File:     frontend/components/ui/ConfirmDialog.tsx — line 86
  Issue:    The "Delete" button has `autoFocus`, meaning keyboard users who press
            Enter immediately after the dialog opens will confirm deletion. Standard
            HIG/WCAG guidance places `autoFocus` on the safer action (Cancel) in
            destructive dialogs.
  Fix:      Move `autoFocus` from the confirm button to the Cancel button.

[L-3] ProfileCard delete swallows errors silently (Bug / UX)
  File:     frontend/components/ProfileList.tsx — lines 116-119
  Issue:    `handleDelete` (inside `ProfileCard`) calls `store.deleteProfile` and
            notifies the parent via `onDelete?.()` on success. If the deletion fails,
            `err` is returned from `SwipeToDelete.handleConfirm` (which shows it in
            the delete zone), but the `ProfileCard.handleDelete` itself only checks
            for falsy error — it does not surface any error to the user because
            `SwipeToDelete` passes the return value of `onDelete()` to its own
            `handleConfirm`, which does call `setIsPending(false)` and
            `setShowConfirm(false)` regardless of the error. The user sees the dialog
            close with no feedback.
  Fix:      Have `ProfileCard.handleDelete` return the error string from the store
            so `SwipeToDelete.handleConfirm` can display it. The `SwipeToDelete`
            `onDelete` prop is already typed as `() => unknown`, so changing it to
            `() => Promise<string>` (matching `ProfileTabs`) would align the pattern.

[L-4] BirthdayField local state does not sync when birthday prop changes (Bug)
  File:     frontend/components/ProfileTabs.tsx — lines 313-314
  Issue:    `const [value, setValue] = useState(birthday ?? "")` initialises from
            the prop but never re-syncs. If the parent re-fetches the profile (e.g.
            after `onSaved`), the `birthday` prop updates but `value` stays stale.
            When the user reopens the edit input it still shows the old value.
  Fix:      Add a useEffect:
              useEffect(() => { setValue(birthday ?? ""); }, [birthday]);

[L-5] StoreProvider migration check parses localStorage without error handling (Bug)
  File:     frontend/lib/store/StoreProvider.tsx — lines 107-113
  Issue:    `JSON.parse(localStorage.getItem("orbits:profiles") ?? "[]")` is called
            without try/catch. If localStorage contains malformed JSON (e.g. from a
            prior crash or a different app writing to the same key), this throws an
            uncaught exception that breaks the entire auth state detection effect.
  Fix:      Wrap in try/catch: if parsing fails, treat as empty array and do not
            show the migration modal.

[L-6] getInitials crashes on empty name (Bug)
  File:     frontend/lib/formatters.ts — lines 13-17
  Issue:    `getInitials("")` → `parts = [""]` → `parts[0][0]` is `undefined` →
            returns "UNDEFINED" when `.toUpperCase()` is applied. A profile with an
            empty `full_name` would crash the initials avatar render.
  Fix:      Add a guard at the top:
              if (!name.trim()) return "?";

[L-7] relativeTime does not handle future dates (Bug / UX)
  File:     frontend/lib/formatters.ts — lines 40-53
  Issue:    When `dateStr` is in the future, `diffMs` is negative, `minutes` is
            negative, and the function returns "just now" (because `minutes < 1`).
            An event date set one week from now would display "just now" in the
            "Last edit" timestamp. The function also has no "years" bucket — anything
            older than ~4 weeks is shown as "X months ago" forever.
  Fix:      (a) Add a future check: `if (diffMs < 0) return "in the future"` or
                compute forward-looking strings ("in X days").
            (b) Add a years bucket:
                  const years = Math.floor(days / 365);
                  if (years >= 1) return `${years} year${years !== 1 ? "s" : ""} ago`;

[L-8] ProfileList "Last edit" label uses created_at, not any edit timestamp (UX)
  File:     frontend/components/ProfileList.tsx — line 157
  Issue:    The card displays "Last edit {relativeTime(profile.created_at)}" but
            `created_at` is set once at creation and never updated. The label is
            misleading — it always shows when the person was added, not when they
            were last edited.
  Fix:      Either (a) add an `updated_at` column to `profiles` and update it on
            every mutation, or (b) change the label to "Added" to be accurate.

[L-9] updateBirthday in SupabaseDataStore missing user_id defense-in-depth (Security)
  File:     frontend/lib/store/SupabaseDataStore.ts — lines 203-213
  Issue:    `updateBirthday` filters only by profile `id`, without also checking
            `user_id`. RLS protects this, but unlike createProfile/createFact/etc.
            there is no defense-in-depth user_id filter on the update query.
  Fix:      Fetch `userId` via `this.getUserId()` and add `.eq("user_id", userId)`.

[L-10] auth.ts signIn/signUp do not validate email or password format server-side (Security)
  File:     backend/actions/auth.ts — lines 10-44
  Issue:    Email and password are read from FormData and passed directly to Supabase
            Auth with no server-side validation (no Zod schema). While Supabase will
            reject invalid credentials, an empty email or a single-character password
            will result in a Supabase error message exposed directly to the client
            rather than a friendly validation message.
  Fix:      Add a Zod schema:
              const AuthSchema = z.object({
                email: z.string().email("Invalid email"),
                password: z.string().min(6, "Password must be at least 6 characters"),
              });
            Parse before calling Supabase.

[L-11] createCard does not validate custom_fields in the server action (Security)
  File:     backend/actions/cards.ts — lines 12-19
  Issue:    `CreateCardSchema` does not include `custom_fields`. However the
            `SupabaseDataStore.createShareableCard` passes `custom_fields` directly
            to Supabase (line 537). The server action path ignores custom_fields
            entirely, so shareable cards created via server action will always have
            null custom_fields even if the user intends to set them.
  Fix:      Add `custom_fields` to `CreateCardSchema` with appropriate validation,
            or document clearly that the server action intentionally omits them.

[L-12] ProfileClient re-fetches all data (profile + facts + events) on every mutation (Performance)
  File:     frontend/components/ProfileClient.tsx — lines 63-79
  Issue:    `onMutate` is wired to `loadAll`, which fires three parallel Supabase
            queries every time a single fact or event is added or deleted. After
            adding one note, the entire profile, all facts, and all events are
            re-fetched. For profiles with many facts/events this is wasteful.
  Fix:      Split into `loadProfile`, `loadFacts`, `loadEvents` and pass the
            specific one to each child (ProfileTabs already distinguishes between
            fact mutations and event mutations). Alternatively, use SWR or
            React Query for more granular cache invalidation.

[L-13] DashboardClient does not show an error state if getProfiles throws (Bug / UX)
  File:     frontend/components/DashboardClient.tsx — lines 24-28
  Issue:    `loadProfiles` calls `store.getProfiles()` without a try/catch. If
            Supabase is unavailable or returns an error, `SupabaseDataStore.getProfiles`
            throws (line 60-61 in SupabaseDataStore.ts). The exception propagates
            uncaught to React, which would show a generic error boundary (if one
            exists) or a white screen.
  Fix:      Wrap `loadProfiles` in try/catch and set an `error` state that renders
            a user-friendly message (e.g. "Could not load your contacts. Please
            refresh.").

[L-14] addProfileForm uses window.confirm for duplicate detection (UX)
  File:     frontend/components/AddProfileForm.tsx — lines 32-40 (via AddPersonPanel)
  Issue:    The duplicate name check uses `window.confirm()`, which is blocked in
            some browsers (e.g. iframes), is not styleable, and is jarring on mobile.
            The rest of the codebase uses the custom `ConfirmDialog` component.
  Fix:      Replace `window.confirm()` with the app's own `ConfirmDialog` component,
            consistent with SwipeToDelete and DeleteProfileButton.

[L-15] sendFriendRequest in SupabaseDataStore vs. server action use different status values
  File:     frontend/lib/store/SupabaseDataStore.ts — line 311
            backend/actions/friends.ts — line 41
  Issue:    The client store's `sendFriendRequest` (Orbit Code flow) inserts with
            `status: "accepted"` immediately. The server action inserts with
            `status: "pending"` and requires an accept step. These two flows create
            friendships with different initial states, which could confuse the UI
            if both paths are ever triggered for the same feature.
  Fix:      If the Orbit Code flow is intentionally auto-accepted, document this
            discrepancy clearly. Consider unifying into a single flow or renaming
            the method to `connectViaOrbitCode` to make the difference explicit.
